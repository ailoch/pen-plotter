# SVG-to-Gcode Pen Plotter

Converts an SVG drawing into G-code that drives a Bambu Lab P1S 3D printer as an XY pen plotter.

## Running

[`_Process.py`](_Process.py) is the entry point — prompts for SVG input and gcode output files, then runs:
1. **Parse** SVG → geometry model
2. **Stroke** — generate multi-pass strokes from each shape's stroke width/style
3. **Infill** — add concentric loops to fill interiors of shapes
4. **Drop raw geometry** — remove the now-unneeded parsed centerlines
5. **Route** — reorder paths to minimize travel distance
6. **Generate** — converts intenal geometry model to G-code (lines → G1, arcs → G2/G3)

All logic lives in `lib/`, importable for testing (no module-level side effects).

## Geometry Model

[`lib/geometry.py`](lib/geometry.py): `Segment` (Line/Arc/QuadraticBezier/CubicBezier) → `Path` (list of segments plus a `lineType: LineType`, a subpath) → `PathObject` (list of Path + Style + Transform). `Document` = list of PathObjects. `lineType` (from `lib/settings.py`'s `LineType` enum — `RAW_GEOMETRY`/`STROKE`/`INFILL`/`GAP_INFILL`/`TRAVEL`/bounds-debug values) tags which draw role a subpath belongs to; it lives on `Path` rather than `Style` because a single `PathObject` mixes roles (stroke passes vs. infill loops vs. gap-fill strokes) once stroke/infill generation runs. Defaults to `RAW_GEOMETRY` — the un-stroked, un-filled centerline geometry parsed straight from the SVG, which is a *source* for stroke/fill generation rather than something drawn directly (it has no `heights`/`speeds`/`accels`/`shortTravelThresholds` entry of its own, since nothing should ever emit it as gcode); `tessellate()` carries whatever `lineType` a `Path` has through onto the returned `Path`. `lib/stroke.py`'s `dropRawGeometry` removes every `RAW_GEOMETRY` subpath (and any object left with none) right after stroke+infill generation, before routing ever runs — `lib/plot.py`'s `_addPath` also skips the role defensively, but by construction should never see one.

`Style` also carries stroke info: `strokeWidth`, `strokeColor` (`None` means no stroke — the SVG default), `linejoin`/`linecap`/`miterlimit`, and `dasharray` (`None` means solid; parsed but not yet consumed — dash generation is future work). `strokeWidth`/`dasharray` are in SVG user units at parse time; `PathObject.applyTransformations()` scales both by `sqrt(|det|)` of the applied transform's 2x2 part (the uniform-scale equivalent — a single width/length can't represent true non-uniform scaling) at the same time it transforms the segments, so they stay correct after the printer-space transform is applied.

Key: `Path.point(t)` spans whole subpath (0 ≤ t ≤ 1), `isClosed()`, `isFillable()` (encloses area, used to gate infill — separate from closed state), `tessellate(tolerance, allowArcs)` reduces to Line/Arc within tolerance, `rotateTo(index)` re-anchors closed paths (for routing).

## Key Pipeline Stages

**Parse** ([`lib/svgparse.py`](lib/svgparse.py)): `loadSvg` reads the file into an svgelements tree (raising `SvgParseError` on failure); `parseSvg` then turns it into a `Document` of `PathObject`s — handles groups, paths, rects (including rounded corners), circles/ellipses, lines/polylines/polygons; applies transform stack (flip Y for gcode, offset for pen geometry). `Rect`/`Path`/`SimpleLine`/`Polyline`/`Polygon` are handled by one shared branch in `parseSvgElement`, since all five expose their geometry as the same Move/Line/Arc/QuadraticBezier/CubicBezier/Close segment vocabulary via `segments(transformed=False)` — `transformed=False` because each element's own `transform` attribute is already baked into its geometry by svgelements' structural parser (confirmed: `node.transform` reads back as identity), so re-applying it would double-transform; only the ancestor group chain (`docTransform`) still needs composing into `nodeTransform`. `Circle`/`Ellipse` are deliberately handled separately, not through that shared branch: their `segments()` decomposes the shape into 4 quarter-`Arc`s + a closing `Line`, whereas a single `Arc` (default `sweep=2*pi`) already represents the whole shape exactly in one segment — unifying would trade 1 segment for 5 with no feature gained (unlike `Rect`, there's no `rx`/`ry`-style extra attribute to pick up). `<use>` remains unhandled. If the SVG viewport size doesn't match `canvasSize`, `promptRescale` asks the user how to reconcile it (see Alignment & Scaling below); the drawing is then centered on the canvas. `loadSvg` and `promptRescale` are split out of `parseSvg` so `_Process.py` can run them (file I/O + the interactive prompt) before starting the timer/profiler that measures only the computational parse.

**Stroke** ([`lib/stroke.py`](lib/stroke.py)): For each `PathObject` with a set `style.strokeColor` and `strokeWidth > 0`, offsets every `RAW_GEOMETRY` subpath outward with pyclipper to build a family of concentric `STROKE` passes — structured like `lib/infill.py` (module-level clipper helpers, warn-and-skip on `ClipperException`), and in fact reuses several of them directly (`_SCALE`/`_toClipperPath`/`_appendLoop`/`_drawArcTolerance`/`_joinType`, imported from `lib.infill`). The naive approach (one pass per `fillSpacing`, all offset outward from the centerline) would leave the *centerline itself* unstroked and skew the passes to one side; instead: `numPasses = max(1, ceil(strokeWidth / fillSpacing))` (forced to 1 if `fillSpacing <= 0`) and `s = strokeWidth / numPasses` is the actual pitch between passes (≤ `fillSpacing`). Odd `numPasses` gets a center pass — an exact `copy.deepcopy` of the raw subpath (preserves arcs/beziers exactly; this is the *whole* story for a hairline stroke ≤ `fillSpacing`, keeping a single centerline pass) — plus rings at `k*s` for `k = 1..(numPasses-1)/2`; even `numPasses` has no center pass, with rings at `(k-0.5)*s` for `k = 1..numPasses/2` instead (`_passDeltas`). Either parity lands the outermost ring's `delta + s/2` at exactly `strokeWidth/2`, so the outermost pass's ink reaches the true stroke edge. Each ring offsets a flattened polyline (`tessellate(tolerance, allowArcs=False).vertices()` — closed subpaths omit the duplicate end point, open ones include it, matching pyclipper's `AddPath` expectation) with `MiterLimit = style.miterlimit` and a join type from `style.linejoin` (`miter → JT_MITER`, `round → JT_ROUND`, `bevel → JT_SQUARE`, via `_joinType`); a closed subpath offsets as `ET_CLOSEDLINE`, whose `+delta` Execute yields *both* the outer ring and the inner hole in one call (2 contours per ring), while an open subpath offsets per `style.linecap` (`butt/round/square → ET_OPENBUTT/ET_OPENROUND/ET_OPENSQUARE`), whose `+delta` Execute yields one contour wrapping both sides plus caps. Every resulting contour is appended via the shared `_appendLoop`, tagged `STROKE`. If pyclipper isn't installed, or `machine.generateStroke` is set `false`, `generateStroke` skips the multi-pass expansion entirely and falls back to a single center-pass `deepcopy` per stroked subpath (the pre-multi-pass centerline-only look) - the missing-pyclipper case additionally warns once. Must run before `generateInfill`, since a stroked object's fill inset depends on `strokeWidth`/`linejoin` (see below).

**Infill** ([`lib/infill.py`](lib/infill.py)): For each fillable `PathObject`, use pyclipper to generate concentric inward-offset loops respecting `fill-rule` (`nonzero`/`evenodd`). Offset the same pyclipper paths repeatedly (`_concentricLoops`) to avoid compounding drift. Re-fit loop polylines with `fitLines=True` to recover circular arcs. Gated on `lineType == RAW_GEOMETRY` (so a previous pass's own generated loops are never re-used as a fill source) plus `isFillable()` (encloses area, computed via shoelace formula), not `isClosed()`. A stroked object's fill must stay clear of the stroke band and follow its inner edge: the first ring's inset is `strokeWidth/2 + spacing/2` (instead of the unstroked `spacing/2`), and `_concentricLoops` takes an optional `joinType` (default `JT_ROUND`) so that first ring can be offset with the stroke's own `linejoin` (`_joinType`, shared with `lib/stroke.py`) rather than always rounding corners. Then a **gap-fill pass** (`_gapFill`, on by default via `processing.generateGapInfill`) closes the two gaps concentric loops leave — acute-corner wedges and fractional-width slivers (where loops closing in from both sides meet with a residual `<spacing` gap). It's the slicer "gap infill" idea: stroke every drawn centerline by `spacing/2` via `ET_CLOSEDLINE` to get the covered region, `CT_DIFFERENCE` it against the fill region to get the uncovered residue, clean it (a morphological open at `tolerance/2` plus an `(spacing/2)²` min-area threshold — which also discards the benign sub-`penWidth` dots between adjacent loops, since `fillSpacing` is deliberately set under the real pen width so those dots are inked anyway), then fill each surviving piece — grouped with its holes via an `Execute2` PolyTree so annular gaps stay whole — with `_fillGap`, whose first inset halves from `spacing/2` down toward `tolerance/2` so a gap narrower than `spacing` still yields a centerline stroke. "Every drawn centerline" here means this object's own `STROKE` passes (flattened the same way as everywhere else) plus the fill loops just generated — never the raw outline itself, since `RAW_GEOMETRY` is never drawn; an unstroked fill-only shape's coverage is therefore just its fill loops. Known limitation: an open subpath's center-pass stroke still offsets as `ET_CLOSEDLINE` here (same treatment as every other centerline), so it's read as closed for coverage purposes — this only matters for an object mixing an open stroked subpath with a separately-fillable closed one, and only overstates coverage (never creates a false gap). The drawn loops use `JT_ROUND` by default; the detection-only offsets (`_offsetPolys`) use the cheaper `JT_SQUARE` (negligible at nozzle scale). Coverage width is `fillSpacing` (the loop pitch), never `visualization.penWidth`. Each generated loop is tagged via `_appendLoop` (shared with `lib/stroke.py`) with its `LineType` — `INFILL` for concentric loops, `GAP_INFILL` for gap-fill strokes.

**Route** ([`lib/route.py`](lib/route.py)): TSP-like optimization (nearest-neighbor + 2-opt + anchor-optimization + rendezvous move) to reorder objects and choose entry points, minimizing pen travel. Two-pass: order internal subpaths within each multi-subpath object, then order objects themselves. Travel cost isn't pure XY distance: `_travelCost` adds an estimated pen lift/lower time (converted to equivalent XY distance at travel speed, so it stays additive with the distance term), mirroring the real pen-up move `lib/plot.py` will later emit - prevMoveHeight → travelHeight on lift, then travelHeight → nextMoveHeight on lower, at `min(speeds[travel], maxVerticalSpeed)` since a printer's Z axis is typically slower than X/Y. No penalty is added when the travel is under `shortTravelThresholds` (the smaller of the departure/arrival roles', same rule `plot.py` uses to decide the pen never lifts at all). A subpath item's role is its single `lineType`; a whole-object item takes its first/last subpath's `lineType` (`_itemLineTypeStart`/`_itemLineTypeEnd`), which can differ when the object mixes roles - `startRole`/`endRole` swap with `rev` the same way `startPt`/`endPt` do. By the time routing runs, `dropRawGeometry` has already removed every `RAW_GEOMETRY` subpath (see Geometry Model), so every item these helpers see has a real drawn role with a `shortTravelThresholds`/`heights` entry.

**Generate** ([`lib/plot.py`](lib/plot.py)): For each path, tessellate to Line/Arc, emit G1/G2/G3. `_addPath` skips a `RAW_GEOMETRY` path defensively if one somehow reaches this stage (it shouldn't - `dropRawGeometry` removes them all beforehand). Intersperse travel moves (short = pen-down, long = pen-up). Each path's `lineType` drives which height/speed/accel and slicer feature label is used for its draw moves (and for the short-travel move immediately before them). Cosmetic: alternate +0.001mm Z per object for preview layering. The prefix/suffix gcode templates are copied through `_fileAppend`, which substitutes `{...}` blocks: each block's contents are an arithmetic expression evaluated against the `replace` namespace — `TRAVEL_HEIGHT`/`TRAVEL_SPEED`/`TRAVEL_ACCEL`/`LINE_WIDTH`/`LOAD_DELAY`/`END_X`/`END_Y` (raw floats) plus the string-valued `EXTRUDER_OFFSET`/`BED_EXCLUDE_AREA`. So `{TRAVEL_SPEED/2}` or `{TRAVEL_HEIGHT + 10}` work, not just a bare `{TRAVEL_SPEED}`; numeric results are formatted via `_fmtNum`, strings pass through when referenced bare. Evaluation is **not** `eval` — it's an AST whitelist (`_evalTemplateNode`) that permits only names (looked up in `replace`), numeric literals, and `+ - * / // %` (with operands required numeric via `_requireNumber`). Deliberately excluded: attribute access, calls, subscripts, and `**`. This matters because a prefix/suffix template can arrive alongside a shared/downloaded config, and `eval` — even with `__builtins__` stripped — is escapable to arbitrary code via attribute traversal from any object literal (`().__class__.__bases__[0].__subclasses__()`); the numeric-operand rule also blocks the `'x'*999999999` / `2**999999999` resource-exhaustion vectors. A block that fails to evaluate (unknown name, bad syntax, arithmetic on a string, or any disallowed construct) is left in place verbatim with a warning, preserving the old leave-unrecognized-keys-as-is behavior.

## Tessellation (`Path.tessellate`)

Bidirectional greedy fitter: reduces any curve to Line/Arc within tolerance, working in normalized subpath-space (0 ≤ t ≤ 1) so fits can span segment boundaries. Key behaviors:
- **Already-final segments atomic**: `Line` passthrough (unless `fitLines=True`), circular `Arc` passthrough → prformance gain.
- **Line merging**: `appendMerging` catches exact-collinear runs the fitter misses (each already fits individually).
- **`fitLines=True`** (used by infill arc-recovery): re-fits raw polygon points → arcs, with optimized fast path for all-`Line` ranges (no per-segment interior sampling).
- **Numerically unstable circumcircles rejected** via `MAX_RADIUS_TO_CHORD` guard — filters near-collinear noise.

## Error Handling

**Missing/invalid config file** — `Settings.initFromJson` catches a missing file (`FileNotFoundError`) and parse errors (JSON syntax, type mismatches) separately, prints a clean one-line error message (e.g., `No terminal defined for 'f' at line 21 col 1`), and proceeds with defaults either way. The program continues so the user can still test with the default settings if needed.

**Invalid SVG file** — `loadSvg` wraps `svgelements` and re-raises any parse error as `SvgParseError`; it also raises `SvgParseError` itself if the parsed SVG has no `viewBox` (this converter needs one to size the drawing — `svg.viewbox` is otherwise `None`, which `promptRescale`/`parseSvg` each re-assert non-`None` since that narrowing doesn't cross function boundaries), or if the `viewBox` has a non-positive width/height (`svgelements` stores negative values as-is rather than rejecting them, which would otherwise silently corrupt the transform math in `parseSvg`). `_Process.py`'s `run()` catches `SvgParseError` around just the `loadSvg` call, prints it, and returns `RunResult.BAD_INPUT`; the main loop re-prompts for a new input file on that result, clears the cached `document`, and retries. No config reload needed.

**Empty/degenerate paths** — a `<path>` with no drawable segments (e.g. `d=""`, or only `Move` commands) is dropped in `parseSvgElement` rather than added to the `Document` with a placeholder empty `Path`: nothing to draw means nothing to route, and `PathObject.start()`/`end()` (used by `route.py`'s ordering) assume at least one segment.

**Output retries & the cached document** — `run()` parses/infills/routes once and caches the resulting `document` in a module global. A write failure (`createFile` returns `False` → `RunResult.BAD_OUTPUT`, e.g. the target gcode is open in another program) re-prompts only for the output file and redoes just `createFile` against the cached `document` — it does not re-parse or re-ask the rescale question. The cache is cleared only when the input file changes (`BAD_INPUT`). The rescale prompt and SVG load run before the timer/profiler starts, so neither the reported time nor the profile includes user think-time.

**Output file safety** — `lib/plot.py`'s `createFile` writes to a temp file in the output directory and only `os.replace()`s it over the real target on full success. If anything fails mid-pipeline (missing settings keys, missing prefix/suffix gcode files, etc.), the output file is left untouched — can't be truncated by a partial crash.

## Settings (`lib/settings.py`, `config/bambu_p1s_config.json`)

`Settings` (a dataclass, in `lib/settings.py` alongside the `LineType` enum it keys heights/speeds/accels/lineTypes by) is shared across the whole pipeline — `parseSvg`, `generateStroke`, `generateInfill`, `orderPaths`, and `createFile` all take a `Settings` instance and read the fields they need directly, rather than being passed individual values. `LineType` has three draw roles (`STROKE`, `INFILL`, `GAP_INFILL`) plus `TRAVEL`, `RAW_GEOMETRY` (see Geometry Model), and three bounds-debug values.

One config file per printer, named `config/<printer>_config.json` (currently just `bambu_p1s_config.json`). Loaded via `commentjson` (supports `//` comments). `machine` (startPos/penOffset/plateSize/safeZoneSize/safeZoneOffset/canvasSize/canvasOffset — see Alignment & Scaling below), `processing` (what is drawn on paper: tessellationTolerance, fillSpacing, prefix/suffix gcode template paths), `motion` (how the pen moves while drawing: per-role heights/speeds/accels/shortTravelThresholds, loadDelay), `visualization` (pen width, cosmetic layering/coloring for Bambu Studio preview), `debug` (showBoundingBoxes, optimizePathOrder, profiling). All fields are type-checked before use; on mismatch, the setting is skipped and a warning is printed. In `motion.heights`/`speeds`/`accels`/`shortTravelThresholds` (and `visualization.lineTypes`), a `"draw"` key sets all three draw roles at once; an explicit `stroke`/`infill`/`gapInfill` key overrides `"draw"` for just that role — this expansion happens once in `initFromJson`, so the rest of the pipeline just indexes by concrete `LineType`. `shortTravelThresholds` is keyed by draw move `LineType` (a travel move shorter than the threshold stays pen-down) — `plot.py`'s `_penMove` looks up the upcoming (arrival) role via `lineType or LineType.STROKE`, same fallback `heights`/`speeds`/`accels` use. When the travel follows a different-role draw move, the threshold is the min of the departing and arriving roles' values, so a short hop into (or out of) a role with a tighter threshold isn't allowed to hide behind the other role's laxer one; `_DrawState.lastLineType` tracks the most recently drawn role to make this comparison. `initFromJson` ends with `_validate()`, which calls `_validateBounds()` (warn-and-continue, never resets to defaults — see Alignment & Scaling) and also warns if `generateGapInfill` is enabled while `fillSpacing <= 0` disables fill entirely. `fillSpacing` doubles as both the infill loop pitch and the stroke pass pitch (see Stroke above).

Positions (`endPos`, `penOffset`, `plateSize`, `safeZoneSize`, `safeZoneOffset`, `canvasSize`, `canvasOffset`) are stored as `complex`, matching how positions are represented everywhere else in the codebase — JSON's 2-element lists are converted via `complex(x, y)` in `initFromJson`. `startPos` is the one exception, kept as a `dict[str, float]` (`{"X":.., "Y":.., "Z":..}`) since it needs a Z component and `createFile`'s per-file `_DrawState.pos` (current nozzle position) is built directly from it.

## Alignment & Scaling (`lib/settings.py`, `lib/svgparse.py`)

All positions/sizes below are in mm, lower-left-corner convention, and nest: **plate ⊇ safe zone ⊇ canvas**.

- `plateSize` — the heatbed rect; its lower-left corner is fixed at the origin. This is the one physical/nozzle-space rect (the bed doesn't care about the pen).
- `safeZoneSize` / `safeZoneOffset` / `safeZoneAlignment` — the rect the pen can move within without colliding with anything. `safeZoneOffset` is in **pen space** (where the pen tip should be), not gcode/nozzle space.
- `canvasSize` / `canvasOffset` / `canvasAlignment` — the paper/drawable surface, also in pen space; `canvasSize` is the field users change to resize the paper.

`safeZoneAlignment`/`canvasAlignment` (`"BL"`/`"BR"`/`"TL"`/`"TR"`, default `"BL"`) name which **plate** corner `safeZoneOffset`/`canvasOffset` is measured from: first char is Bottom/Top, second is Left/Right. The offset is the distance to translate *towards the plate's center* from that corner being coincident with the matching plate corner — e.g. `"BR"` with offset `(2, 1)` starts the rect's bottom-right corner coincident with the plate's bottom-right corner, then shifts it 2mm left and 1mm up. `"BL"` is a no-op under this scheme (shifting right/up from the origin is exactly today's lower-left-offset behavior), so it's the default and existing configs are unaffected. `Settings.initFromJson` resolves both fields into the plain lower-left-corner offset the rest of the pipeline already expects (`_alignedOffset`, called right after the settings loop, before `_validate()`) — nothing downstream of `initFromJson` (routing, bounds checks, svgparse) is alignment-aware; it only ever sees a lower-left offset from the plate origin, same as before this feature existed.

Since gcode X/Y commands the nozzle, and the pen tip sits at `nozzle + penOffset`, a pen-space position `P` corresponds to nozzle position `N = P - penOffset`. `parseSvg`'s printer-space transform (`lib/svgparse.py`) builds its translation this way: it centers the drawing on `canvasOffset + canvasSize/2` (a pen-space point) and only then subtracts `penOffset` to get the nozzle-space translation actually written to gcode.

`Settings._validateBounds()` (called at the end of `initFromJson`) warns — but does **not** alter the loaded values or fall back to defaults — if any containment in the chain doesn't hold, checked in hierarchy order:
- safe zone not fully inside the plate — `safeZoneOffset` is already pen-space (i.e. already expressed in the same physical bed-frame numbers the plate rect uses), so this is a direct compare.
- canvas not fully inside the safe zone (both already pen-space, compared directly).
- nozzle movement not fully inside the plate — the nozzle's actual gcode movement, driving the pen across the safe zone, sits at `safeZoneOffset - penOffset`. A safe zone that keeps the pen on the plate can still walk the nozzle off it, or vice versa, so both are checked.

`_Process.py` calls `promptRescale(svg, settings)` (a thin wrapper over `_promptRescale(svgWidth, svgHeight, canvasSize)`), which returns `(1, 1)` with no prompt if the SVG viewport already matches `canvasSize`; the chosen scale is then passed into `parseSvg`. Otherwise it prints both sizes and prompts for how to reconcile them: keep as-is, fit width, fit height, or stretch to fill both axes. When the canvas and viewport share an aspect ratio, fit-width/fit-height/stretch all reduce to the same scale, so the prompt collapses to just "keep as-is" vs. a single "rescale to fit" — using the same letter (`b`) as "stretch to fill both axes" in the general prompt, since they're the same operation in that case (helps muscle memory). The resulting scale is applied together with the Y-flip, canvas-centering, and `penOffset` compensation as one transform matrix per object (no separate steps).

## Hardware

**Bambu Lab P1S**: E-axis represents pen-tip distance (not filament), `M221 S0` disables extrusion. P1S firmware bug (repro in `bug.gcode`): the planner throttles XY speed to the E-axis limits based on the *raw commanded* E, ignoring the `M221 S0` flow ratio — and it clamps/ignores `M203 E` raises, so the limit can't be lifted from gcode. Workaround: `motion.eAxisMultiplier` (0.05 in the P1S config, default 1.0) scales every emitted E value in `plot.py`'s two E-emitting sites (G1 draws and G2/G3 arcs), keeping the required E rate under the firmware cap; side effect is the slicer's total-filament stat scales down by the same factor (preview rendering is unaffected — it only needs E > 0). `_validate()` warns if the multiplier is ≤ 0, which would drop E from every draw move (falsy args are elided in `_addLine`) and make the slicer render the whole drawing as travels. Pen offset from nozzle via `penOffset` (applied after SVG Y-flip). Renderer clips to `[-5000, 5256]`. Cosmetic `; FEATURE:` comments drive preview coloring.

`lib/plot.py`'s `BED_EXCLUDE_AREA` is built by `_bedExcludeArea` (plate rect minus the canvas rect) for the slicer to render the drawable area. The canvas is placed per `canvasOffset` — in pen space when `showPenPos` (the slicer applies `EXTRUDER_OFFSET = penOffset`), else shifted by `-penOffset` into nozzle space. Since both rects are axis-aligned, the shape is classified directly from which of the 4 plate edges (bottom/right/top/left) the canvas fails to touch ("gap" flags), no polygon library needed: 0 gaps → nothing excluded; all 4 → a ring (`_bridgeContours` keyholes the outer plate loop to the inner canvas hole); 2 opposite gaps → two disjoint strips (also `_bridgeContours`-joined); a contiguous run of 1-3 gaps → `_perimeterWalk` traces a single simple polygon (|/L/C) directly, no seam needed. `_removeRedundantPoints` then drops collinear midpoints from all cases. `safeZone` is not part of this polygon.

## Dependencies

`pyclipper` (infill polygon offsetting), `svgelements` (SVG parsing), `commentjson` (JSONC), `scipy` (arc-length integration). `pyclipper` has no type stubs; `typings/pyclipper/__init__.pyi` is hand-cleaned via `stubgen`.
